Day 22 — Advanced Lab Kickoff & Environment Upgrade

Goal: Set up the upgraded lab environment for the next advanced simulation cycle, expanding beyond the Day 1–21 Windows hacking bootcamp by integrating larger enterprise-scale elements, cloud or hybrid infrastructure, and new adversary emulation capabilities.

Step 1 — Lab Expansion Planning

  • Define the scope for the upgraded environment:
    • Number of hosts (Windows, Linux, cloud)
    • Network segmentation (internal, DMZ, cloud connectivity)
    • Active Directory forest/domain complexity
  • Map objectives for the new training cycle:
    • More complex multi-stage intrusions
    • Multi-adversary simultaneous campaigns
    • Cross-platform attack vectors

Step 2 — Infrastructure Upgrades

  • Add new systems:
    • At least one Linux server for cross-platform targeting
    • A cloud-connected Windows Server (Azure or AWS integration)
    • Workstations with varied OS versions for realistic diversity
  • Implement VLANs or virtual networks to simulate enterprise segmentation.

Step 3 — Cloud Integration

  • Deploy a small cloud lab:
    • Azure Active Directory or hybrid AD sync
    • Cloud-hosted Windows and Linux VMs
    • Simulated SaaS applications for attack surfaces
  • Prepare cloud monitoring tools (Azure Sentinel, AWS GuardDuty, etc.).

Step 4 — New Adversary Emulation Playbooks

  • Select additional threat actors from MITRE ATT&CK to target cloud and hybrid environments.
  • Build red team playbooks for: Cloud credential theft, Identity federation attacks, Cloud persistence mechanisms

Step 5 — Detection Enhancements

  • Extend Sysmon and logging policies to new on-prem and cloud systems.
  • Integrate cloud-native logging (Azure Log Analytics, AWS CloudTrail) into central monitoring.
  • Create correlation rules for hybrid attack paths.

Step 6 — Validation & Baseline

  • Perform initial red team tests on new infrastructure.
  • Blue team records baseline activity and validates visibility across all environments.
  • Confirm detection triggers for both on-prem and cloud events.

End of Day 22 Success Criteria:

✅Upgraded lab with on-prem, cloud, and hybrid systems operational
✅Adversary playbooks prepared for advanced scenarios
✅Monitoring and detection extended to all environments
✅Baseline established for the next advanced simulation cycle

Day 23 — Hybrid Attack Chain Exercise

Goal: Simulate a red team intrusion starting in the on-premises environment and pivoting into the connected cloud infrastructure, while the blue team detects and responds across both environments.

Step 1 — Attack Scenario Design (Red Team)

  • Plan an attack chain that begins with:
    • Compromise of an on-prem Windows workstation or server.
    • Enumeration of hybrid AD or cloud-connected assets.
    • Pivot into the cloud environment via: Compromised Azure AD credentials, Token theft or identity federation abuse.
  • Final objective: Access cloud-hosted sensitive data or resources.

Step 2 — Environment Preparation (Blue Team)

  • Confirm monitoring covers: On-prem logs (Sysmon, Security Event Logs, WEF), Cloud logs (Azure Sentinel, AWS GuardDuty, CloudTrail)
  • Set correlation rules for: Hybrid logon patterns, Cross-environment lateral movement, Unusual API calls from cloud admin accounts

Step 3 — Red Team Execution

  • Initial access on-prem (phishing simulation or credential reuse).
  • Enumerate hybrid identity sync configurations.
  • Capture valid cloud credentials or tokens.
  • Access cloud management console or APIs.
  • Perform high-value action (e.g., create cloud admin account, access storage buckets).

Step 4 — Blue Team Detection & Response

  • Detect on-prem compromise indicators: Event ID 4624 (logon) anomalies, Sysmon Event ID 1 and 3 for process/network activity
  • Detect cloud pivot indicators: Azure sign-ins from unusual IPs or devices, Creation of new privileged cloud accounts, Unusual cloud API usage
  • Contain by: Disabling compromised accounts, Revoking cloud sessions and tokens, Isolating affected systems

Step 5 — Debrief

  • Red team: Present full hybrid attack chain, including timeline and techniques.
  • Blue team: Present detection timeline, containment actions, and missed events.
  • Jointly define improvements to: Cross-environment monitoring, Incident response for hybrid threats

End of Day 23 Success Criteria:

✅Successful simulation of an on-prem to cloud pivot
✅Multiple detection points triggered across both environments
✅Documented improvements to hybrid attack detection and response

Day 24 — Cloud-First to On-Prem Pivot Exercise

Goal: Simulate a red team intrusion that begins in the cloud environment and pivots into the on-premises infrastructure, testing blue team detection and response across the reverse hybrid attack path.

Step 1 — Attack Scenario Design (Red Team)

  • Plan an attack chain starting with:
    • Compromise of a cloud-based account (Azure AD or SaaS credentials).
    • Discovery of hybrid AD sync or VPN access methods.
    • Pivot into the on-prem environment via: Leveraging synchronized cloud credentials, Exploiting misconfigured identity federation.
  • Final objective: Access sensitive on-prem servers or Active Directory Domain Controller.

Step 2 — Environment Preparation (Blue Team)

  • Ensure monitoring covers: Cloud logs (Azure Sentinel, AWS GuardDuty, CloudTrail), On-prem logs (Sysmon, Security Event Logs, WEF)
  • Set correlation rules for: Cloud-to-on-prem logon activity, VPN or remote access from compromised cloud accounts, Service account activity anomalies

Step 3 — Red Team Execution

  • Gain access to cloud account credentials (phishing simulation, password spray, token theft).
  • Enumerate cloud services for hybrid connectivity clues.
  • Use VPN, RDP Gateway, or sync credentials to log into on-prem systems.
  • Conduct on-prem reconnaissance to identify high-value targets.
  • Execute privilege escalation or persistence methods locally.

Step 4 — Blue Team Detection & Response

  • Detect cloud compromise indicators: Unusual sign-in locations or devices, Creation of new app registrations or admin roles
  • Detect on-prem pivot indicators: Event ID 4624 with cloud-synced accounts from unusual sources, Sysmon Event ID 3 showing VPN or remote session traffic
  • Contain by: Disabling affected accounts in both cloud and on-prem AD, Terminating active sessions and VPN connections, Isolating on-prem endpoints involved in the intrusion

Step 5 — Debrief

  • Red team: Present the cloud-first attack chain, pivot techniques, and final objectives.
  • Blue team: Present detection points, response actions, and missed events.
  • Jointly identify monitoring improvements for: Cross-environment authentication, Hybrid identity protections

End of Day 24 Success Criteria:

✅Successful simulation of a cloud-to-on-prem pivot
✅Detection points triggered in both environments
✅Documented enhancements to hybrid identity and pivot detection

Day 25 — Simultaneous Bidirectional Hybrid Attack Challenge

Goal: Simulate two coordinated red team intrusions — one starting in the cloud and moving into the on-premises environment, the other starting on-premises and pivoting into the cloud — running concurrently to stress-test blue team detection and response under intersecting hybrid attack paths.

Step 1 — Attack Scenario Design (Red Team)

  • Red Team A (Cloud-first): Compromise a cloud account (Azure AD or SaaS credentials), Discover hybrid AD sync or VPN connections, Pivot down to on-prem servers or Domain Controller.
  • Red Team B (On-prem-first): Compromise an on-prem Windows host or service account, Enumerate hybrid AD sync configurations, Pivot upward to access cloud services or consoles.

Step 2 — Environment Preparation (Blue Team)

  • Ensure monitoring visibility across: Cloud logs (Azure Sentinel, AWS GuardDuty, CloudTrail), On-prem logs (Sysmon, Security Event Logs, WEF)
  • Set correlation rules for: Simultaneous unusual logons in both environments, Multiple pivot attempts across hybrid boundaries, Service account and admin account activity anomalies

Step 3 — Concurrent Red Team Execution

  • Red Team A begins cloud compromise and starts pivoting into on-prem.
  • Red Team B begins on-prem compromise and starts pivoting into cloud.
  • Both teams operate simultaneously, timing key actions to overlap and create confusion.
  • Use both stealth and noisy techniques to complicate blue team analysis.

Step 4 — Blue Team Detection & Response

  • Detect indicators from both attack chains:
    • Cloud: unusual sign-ins, admin role creation, API anomalies
    • On-prem: Event ID 4624 anomalies, Sysmon Event ID 3 network connections, new scheduled tasks or registry changes
  • Manage simultaneous incident handling: Prioritize highest-risk detections first, Contain compromised accounts and sessions in both environments, Isolate affected endpoints and terminate cloud sessions

Step 5 — Debrief

  • Red teams: Present attack chains, pivot techniques, and how they leveraged concurrent timing.
  • Blue team: Present detection timeline, incident prioritization process, and any missed events.
  • Jointly identify improvements for handling multi-front hybrid intrusions.

End of Day 25 Success Criteria:

✅Successful simulation of bidirectional hybrid attacks
✅Detection points triggered across both attack paths
✅Documented improvements for managing complex, concurrent incidents

Day 26 — Multi-Day Persistent Hybrid Threat Simulation

Goal: Simulate a coordinated red team campaign lasting multiple days, with persistent access maintained in both cloud and on-prem environments, gradually expanding the attack footprint to test blue team endurance, detection consistency, and threat hunting capabilities under prolonged compromise.

Step 1 — Campaign Design (Red Team)

  • Establish separate footholds in:
    • Cloud environment via compromised accounts, API tokens, or app registrations.
    • On-prem environment via local admin accounts, scheduled tasks, or WMI subscriptions.
  • Plan gradual escalation over several days:
    • Day 1–2: Reconnaissance and light privilege escalation.
    • Day 3–4: Lateral movement and cross-environment pivots.
    • Day 5+: Persistence reinforcement and simulated data collection.

Step 2 — Blue Team Preparation

  • Confirm logging and alerting systems can handle multi-day event collection.
  • Prepare threat hunting queries for: Long-term persistence detection, Gradual privilege creep, Anomalous cross-environment activity.
  • Establish daily review and response cadence.

Step 3 — Red Team Execution

  • Deploy persistence in both environments using stealthy techniques from previous exercises.
  • Conduct reconnaissance slowly to avoid detection, mixing in normal user-like behavior.
  • Periodically escalate privileges and pivot into new assets, maintaining operational stealth.
  • Simulate periodic "objective actions" such as accessing sensitive files or cloud storage.

Step 4 — Blue Team Detection & Response

  • Perform daily hunts for:
    • Persistence indicators in registry, scheduled tasks, and WMI subscriptions.
    • Cloud anomalies such as unusual sign-ins, API calls, or admin role changes.
    • Hybrid authentication patterns and cross-boundary logons.
  • Contain compromised accounts or systems promptly without disrupting ongoing monitoring.

Step 5 — Campaign Endgame

  • Red team stages a final simulated exfiltration or destructive action.
  • Blue team attempts full containment and eradication of all red team footholds.

Step 6 — Debrief

  • Red team: Present full campaign timeline, including stealth measures and persistence strategies.
  • Blue team: Present detection timeline, daily hunt effectiveness, and final eradication results.
  • Jointly identify endurance-related gaps and improvements.

End of Day 26 Success Criteria:

✅Persistent access maintained by red team for the planned duration
✅Blue team actively hunted and disrupted red team activity throughout
✅Documented improvements to long-term monitoring and incident response strategies

Day 27 — Full Hybrid Eviction Operation

Goal: Conduct a blue team-led operation to locate and eradicate every red team persistence point and access method in both cloud and on-prem environments, while the red team actively attempts to resist removal and re-establish footholds.

Step 1 — Blue Team Preparation

  • Assemble a complete list of suspected compromised accounts, endpoints, and cloud assets.
  • Prepare enhanced hunting queries for:
    • Registry run keys and uncommon autorun locations
    • WMI event subscriptions in root\\subscription
    • Scheduled tasks, especially modified legitimate ones
    • Group Policy logon scripts
    • Active Kerberos tickets (Golden/Silver)
  • Ensure cloud logs (Azure, AWS, SaaS) and on-prem logs (Sysmon, WEF) are accessible.

Step 2 — Red Team Resistance Planning

  • Identify remaining footholds from Day 26.
  • Deploy backup persistence methods that are difficult to detect.
  • Use deceptive tactics to delay blue team removal, such as creating decoy persistence points.

Step 3 — Eviction Execution

  • Blue team methodically hunts for and removes persistence on all systems.
  • Red team attempts to: Re-establish persistence using alternate techniques, Exploit any gaps in detection coverage, Blend malicious actions into normal system activity.
  • Blue team tracks and neutralizes each re-entry attempt.

Step 4 — Containment and Credential Hygiene

  • Reset or disable all accounts suspected of compromise.
  • Invalidate Kerberos tickets by resetting KRBTGT twice.
  • Rotate cloud API keys and service account credentials.

Step 5 — Verification

  • Re-run full persistence hunting queries across both environments.
  • Monitor for any suspicious activity for at least one simulated day after eviction.
  • Confirm no red team-controlled persistence methods remain.

Step 6 — Debrief

  • Blue team: Present full list of persistence methods removed, timeline, and challenges faced.
  • Red team: Share which footholds survived the longest and why.
  • Jointly identify improvements in: Detection coverage, Response speed, Resilience against re-entry attempts

End of Day 27 Success Criteria:

✅All red team persistence methods removed
✅Blue team successfully blocked re-entry attempts
✅Clear documentation of eradication process and updated prevention measures

Day 28 — Post-Eviction Resilience Test

Goal: Simulate a delayed red team comeback after the Day 27 eviction, using new tactics to attempt re-infection of the hybrid environment. Test whether the blue team's updated defenses and procedures can prevent reinfection and maintain a clean security posture.

Step 1 — Red Team Preparation

  • Review detection gaps or slow response areas noted during Day 27.
  • Design new persistence and access methods not used previously:
    • Exploiting overlooked cloud API permissions.
    • Leveraging legitimate third-party integrations.
    • Using non-traditional autorun locations.
  • Plan for low-noise re-entry to avoid triggering immediate alerts.

Step 2 — Blue Team Readiness Check

  • Re-verify that updated detection rules from Day 27 are active.
  • Ensure monitoring is live for: Newly added event IDs or Sysmon filters, Expanded registry and scheduled task coverage, Cloud API and identity event monitoring.
  • Set up a quick-response containment plan for suspected reinfection attempts.

Step 3 — Red Team Re-Entry Attempt

  • Deploy stealth access vectors in both cloud and on-prem.
  • Attempt to create new persistence that mimics legitimate system behavior.
  • Slowly escalate privileges while blending in with normal user activity.

Step 4 — Blue Team Detection & Response

  • Actively hunt for anomalies:
    • Cloud logins from unexpected geolocations or devices.
    • Low-frequency but suspicious process executions.
    • Persistence methods in unusual registry paths or script folders.
  • Respond quickly to any confirmed foothold attempts: Disable affected accounts, Remove malicious persistence, Isolate endpoints.

Step 5 — Verification

  • After red team operations conclude, perform a full hybrid persistence sweep.
  • Cross-check cloud and on-prem systems for missed access methods.
  • Validate that no red team footholds remain.

Step 6 — Debrief

  • Red team: Share which methods were attempted, which were detected, and which (if any) evaded detection.
  • Blue team: Present detection successes, gaps, and response times.
  • Jointly update defenses to address any remaining weaknesses.

End of Day 28 Success Criteria:

✅Blue team prevented or quickly removed all new red team footholds
✅No undetected persistence methods remain after verification
✅Updated defenses reflect lessons learned from post-eviction testing

Day 29 — Long-Game Deception Operation

Goal: Test the blue team's ability to detect and respond to extremely subtle, low-activity red team footholds designed to blend seamlessly with legitimate operations over an extended period.

Step 1 — Red Team Strategy

  • Establish persistence mechanisms that mimic normal user or system behavior:
    • Scheduled tasks disguised as legitimate maintenance jobs.
    • Registry entries under names matching standard OS or application keys.
    • Cloud automation scripts with benign-appearing descriptions.
  • Space out activity to avoid triggering pattern-based alerts.
  • Use legitimate tools (LOLBins) and processes for execution.

Step 2 — Blue Team Preparation

  • Expand anomaly detection baselines to include long-term behavior patterns.
  • Enable advanced logging:
    • Sysmon with detailed process and network event capture.
    • Cloud service activity logs with historical trend analysis.
  • Prepare for multi-week hunts with periodic log reviews.

Step 3 — Red Team Execution

  • Deploy persistence points that activate rarely (e.g., once every few days).
  • Conduct minimal reconnaissance at random intervals.
  • Use data staging methods that appear like standard file transfers or backups.
  • Avoid privilege escalation unless absolutely necessary to maintain access.

Step 4 — Blue Team Threat Hunting

  • Perform scheduled deep-dive hunts into:
    • Rarely used scheduled tasks or registry entries.
    • Cloud automation scripts or policies with recent modifications.
    • Processes that run infrequently but consistently over time.
  • Compare findings against baseline behaviors to identify outliers.

Step 5 — Containment and Verification

  • Isolate suspected persistence mechanisms for further analysis.
  • Remove confirmed malicious footholds while monitoring for fallback methods.
  • Conduct a post-removal watch period to ensure no reactivation occurs.

Step 6 — Debrief

  • Red team: Share concealment techniques, timing patterns, and methods that went undetected.
  • Blue team: Present anomalies found, detection gaps, and time to discovery.
  • Update detection rules to better spot low-frequency, low-noise activity.

End of Day 29 Success Criteria:

✅Blue team identified and eradicated long-term low-activity footholds
✅Detection coverage improved for rare or disguised malicious activity
✅Documented strategies for both defending against and executing long-game deception

Day 30 — Grand Finale Hybrid Breach Campaign

Goal: Conduct a multi-week, multi-vector red team campaign combining every tactic from Days 1–29 into one continuous hybrid breach operation. The blue team must defend against layered deception, persistence, cloud/on-prem pivots, and staged objectives under realistic pressure.

Step 1 — Campaign Planning

  • Red Team:
    • Create a detailed playbook that blends: Initial access (phishing, credential compromise, service exploitation), Lateral movement in both cloud and on-prem environments, Persistence methods from stealthy to noisy, Long-game deception with low-activity footholds.
    • Stage multiple objectives across the timeline.
  • Blue Team:
    • Prepare full-spectrum monitoring for all known tactics.
    • Establish clear escalation and containment procedures.
    • Assign roles for 24/7 simulated coverage during the campaign.

Step 2 — Initial Breach Phase

  • Red team gains footholds in both environments (cloud and on-prem) within the first simulated days.
  • Use different entry vectors for each to mask coordination.
  • Maintain stealth while beginning reconnaissance.

Step 3 — Escalation and Expansion

  • Conduct gradual privilege escalation.
  • Begin selective lateral movement, blending with normal operations.
  • Introduce deception elements to mislead the blue team.

Step 4 — Cross-Environment Pivots

  • Pivot from cloud to on-prem and vice versa, using techniques from Days 23–24.
  • Leverage hybrid identity and sync misconfigurations.

Step 5 — Sustained Operations

  • Keep persistence active in multiple forms: Long-term low-frequency footholds (Day 29), Redundant high-privilege persistence points.
  • Stage simulated exfiltration or destructive actions at planned intervals.

Step 6 — Blue Team Response

  • Actively hunt daily for indicators across both environments.
  • Manage multiple simultaneous alerts.
  • Adapt to changing TTPs throughout the campaign.

Step 7 — Campaign Climax

  • Red team executes a coordinated final push toward high-value objectives.
  • Blue team must prioritize containment and eradicate all red team access.

Step 8 — Final Debrief

  • Red team: Present full campaign timeline, objectives, and evasion tactics.
  • Blue team: Present detection points, response metrics, and lessons learned.
  • Jointly compile a master incident report and updated defense roadmap.

End of Day 30 Success Criteria:

✅Fully executed multi-phase hybrid campaign.
✅Comprehensive blue team detection and response documented.
✅Master incident report completed with actionable defense improvements.
